# Copilot Instructions: Validador de Ideia de Negócio - Versão 3 Avançado

## Project Overview

**Goal:** Build a multi-agent IA architecture for business idea validation using 3 specialized agents (Market, Technical, Financial) that analyze user ideas in parallel and consolidate results in a unified dashboard.

**Version:** Versão 3 - Avançado (Advanced)  
**Team Size:** 5 members  
**Timeline:** 8 phases (Fase 0–7), backend-first, frontend follows  
**Evaluation Criteria:** Semantic commits (all members), IA justification, 7+ tests, ≥30% coverage, complete documentation (PRD, VIABILIDADE, prompts, UML, user stories)

### Backend
- **Runtime:** Node.js 18+ with Express 4.x
- **Database:** SQLite 3 with Prisma ORM
- **IA:** OpenAI API (gpt-4o-mini in dev, gpt-4o for production)

### Shared
- **Versioning:** Git + GitHub with semantic commits
- **Branching:** feature/* → develop → main (protected)

## Code Standards & Conventions

### Conventional Commits
Use semantic prefixes with clear, descriptive messages (all team members must follow):
```
feat(agents): implement mercado agent with JSON validation
fix(orchestrator): handle partial agent failures correctly
docs(VIABILIDADE): add cost/benefit section
test(auth): add JWT middleware tests
chore(deps): update openai package to 4.3.x
refactor(api): consolidate error handling middleware
```

**Format:** `<type>(<scope>): <description>`  
**Types:** feat, fix, docs, test, chore, refactor, style, perf  
**Scope:** Which module (agents, orchestrator, auth, frontend, etc.)  
**Description:** What changed and why (imperative mood, lowercase)  

**Why:** Automatic changelog generation, better git blame/history, rubric compliance for "commits by all members"

### Semantic Branching
```
main/                  # Protected; merges only from develop (phase ends)
  develop/             # Integration branch; daily merges, CI/CD gated
    feature/agents              # Membro B leads; mercado, tecnico, financeiro agents
    feature/orchestrator        # Membro A leads; parallel execution + consolidation
    feature/auth-db             # Membro C leads; Prisma + JWT
    feature/frontend            # Membro D leads; React components + integration
    feature/tests-docs          # Membro E leads; test suite + documentation
    feature/api-documentation   # Swagger/OpenAPI schemas
    bugfix/xyz                  # For hotfixes during phases
    chore/deps                  # Dependency updates
    docs/xyz                    # Documentation-only changes
```

**Naming Convention:**
- `feature/*` - New feature or story
- `bugfix/*` - Production bug fix (merge to develop first, then main)
- `docs/*` - Documentation updates (README, VIABILIDADE, etc.)
- `chore/*` - Dependency updates, build config
- `test/*` - Test-only changes (e.g., improving existing tests)

**Branch Lifecycle:**
1. Create from `develop` with semantic name
2. Keep branches short-lived (<3 days ideal)
3. Push daily to avoid local-only history
4. Delete after merge (GitHub auto-cleanup)

### Pull Requests
**Mandatory checklist in PR description:**
- [ ] Gherkin scenarios written first (BDD requirement)
- [ ] All BDD scenarios passing (`npm run test:bdd`)
- [ ] Unit tests passing (`npm test`)
- [ ] E2E tests passing (`npm run test:e2e`)
- [ ] Coverage report attached; >=30% maintained
- [ ] Gherkin feature file linked (e.g., "See features/validation.feature")
- [ ] No breaking changes to API contract (or explicit contract update)
- [ ] Documentation updated (README, docs/*, comments)
- [ ] Semantic commit messages used

**Approval Rules:**
- Minimum 1 approval required
- Module owner must review (see CODEOWNERS)
- No merge conflicts left unresolved

## Development Workflow

### Daily Standup (15 min)
- **What did you finish yesterday?** (reference your PR)
- **What are you working on today?** (reference task from Phase N)
- **Any blockers?** (API contract broken? Need review?)

### Integration Cadence
1. **Develop Feature** in your branch (write tests alongside)
2. **Open PR** with checklist filled in
3. **Request Review** from module owner (see CODEOWNERS)
4. **Address Feedback** with commits (don't force-push; keeps history clean)
5. **Merge to `develop`** once approved (Membro A can help if conflicts)
6. **Verify in develop** - run full test suite, check no regressions

### Phase Boundaries
- **End of Phase:** Integration test on `develop`
- **Phase Demo:** 15 min walkthrough with acceptance criteria checklist
- **Release to main:** Only at phase end (Membro A coordinates)

### API Contract Breaking Changes
**If you must change the contract:**
1. Open issue: "BREAKING: <reason>"
2. Get approval from Membro A + affected members (frontend/backend)
3. Update **all** implementations simultaneously:
   - Backend response schema
   - Frontend request/response handling
   - Tests for both
   - Documentation (docs/prompts.md if agents change)
4. Merge in single PR (not sequential)

## Prompt Engineering Guidelines

### Agent Response Requirements
All agents **must** respond with **valid JSON only**, no markdown, no extra text.

### Agente de Mercado
```
You are a senior market analyst. Analyze the business idea below and return a JSON with:
{
  "problema": "<precise description of the problem solved>",
  "publico_alvo": { "primario": "<primary audience>", "secundario": "<secondary>" },
  "tam": "<qualitative market size estimate>",
  "concorrentes": ["<competitor 1>", "<competitor 2>"],
  "diferencial": "<what differentiates this idea>"
}
Be objective. Respond ONLY with the JSON, no additional text.
Idea: {ideia}
```

### Agente Técnico
```
You are a software architect. Evaluate the technical feasibility of the idea below and return:
{
  "complexidade": "baixa|média|alta",
  "stack_sugerida": ["<tech 1>", "<tech 2>"],
  "componente_ia": "<how IA would be integrated into the product>",
  "limitacoes_tecnicas": ["<limitation 1>", "<limitation 2>"],
  "mvp_estimativa": "<estimated MVP timeline in weeks>"
}
Be objective. Respond ONLY with JSON.
Idea: {ideia}
```

### Agente Financeiro
```
You are a startup financial analyst. For the idea below, return:
{
  "modelo_receita": "<freemium|subscription|transaction|etc>",
  "custo_operacional_ia": "<qualitative estimate with paid API>",
  "viabilidade_financeira": "baixa|média|alta",
  "proximo_passo": "<concrete action to validate financially>"
}
Be objective. Respond ONLY with JSON.
Idea: {ideia}
```

### Versioning Prompts
When updating a prompt:
1. Document old version in `docs/prompts.md` with date
2. Add **why** it changed (e.g., "added stack_sugerida field to guide MVP")
3. Test with 3-5 real business ideas
4. Update agent implementation + tests simultaneously

## Testing Strategy: BDD + TDD + E2E

### Behavior-Driven Development (BDD) Approach
All features follow BDD: write Gherkin scenarios first, then implement code to pass tests.

**Why BDD:**
- Bridges developers and stakeholders with natural language (Given-When-Then)
- User stories directly become executable acceptance tests
- Single source of truth: Feature → Gherkin → Jest → Code
- Better traceability and documentation

**Test writing order (always):**
1. Define user story and acceptance criteria
2. Write Gherkin scenarios (`.feature` file)
3. Implement step definitions
4. Write Jest/Playwright tests
5. Implement feature code

### Test-Driven Development (TDD) Mandate
**Every coded feature must have BDD scenarios + Jest tests before merge.** No exceptions.

**Definition:** For each feature PR:
- Gherkin scenarios for all acceptance criteria (Given-When-Then)
- Step definitions linking scenarios to Jest tests
- Unit tests for new functions/methods
- Integration tests for API endpoints
- Mock tests for external APIs (OpenAI)
- At least 1 test per acceptance criterion

### Gherkin Feature Files (Given-When-Then)

**Location:** `features/` folder at project root  
**File naming:** `<feature-name>.feature` (e.g., `validate-idea.feature`, `authentication.feature`)

**Example 1: Business Idea Validation**
```gherkin
Feature: Validate Business Idea with Multi-Agent Analysis
  As an entrepreneur
  I want to submit my business idea and receive structured analysis
  So that I can make informed decisions

  Background:
    Given the user is logged in with email "entrepreneur@example.com"
    And the API server is running on http://localhost:3001

  Scenario: Successful idea validation with all 3 agents
    Given the user is on the idea submission form
    When the user enters "A SaaS platform for freelance developers" 
    And the user clicks the "Validar" button
    Then the response should contain all 3 agent analyses (Mercado, Tecnico, Financeiro)
    And the dashboard should display results within 10 seconds
    And the validation should be persisted in the database

  Scenario: Partial agent failure returns consolidated response
    Given the user is on the idea submission form
    And the Técnico agent is temporarily offline
    When the user enters a valid business idea
    And the user submits the form
    Then the response should return 200 (not 500)
    And the response should include error for Técnico agent only
    And the dashboard should show available results with failure notice
```

**Example 2: User Authentication**
```gherkin
Feature: User Authentication with JWT
  As a user
  I want to log in with my email and receive a JWT token
  So that I can access protected endpoints

  Scenario: Successful login
    Given the user is on the login page
    When the user enters "user@example.com" in the email field
    And the user clicks the "Login" button
    Then a POST request is made to /auth/login
    And the response includes token, expiresIn, and userId
    And the token is stored in localStorage
    And the user is redirected to /dashboard

  Scenario: Missing email validation
    Given the user is on the login page
    When the user leaves the email field empty
    And the user clicks the "Login" button
    Then no API request is made
    And the form displays "Email is required"
```

**Step definition example (validation.steps.js):**
```javascript
const { Given, When, Then, Before, After } = require('@cucumber/cucumber');
const axios = require('axios');

Before(function() {
  this.apiClient = axios.create({ baseURL: 'http://localhost:3001' });
  this.authToken = null;
});

When('the user enters {string} in the idea field', async function(idea) {
  this.ideaInput = idea;
});

When('the user clicks the {string} button', async function(buttonText) {
  // Playwright code or API direct call
  const response = await this.apiClient.post('/api/validar', {
    ideia: this.ideaInput
  }, {
    headers: { Authorization: `Bearer ${this.authToken}` }
  });
  this.response = response.data;
});

Then('the response should contain all 3 agent analyses', function() {
  expect(this.response.consolidado.mercado).toBeDefined();
  expect(this.response.consolidado.tecnico).toBeDefined();
  expect(this.response.consolidado.financeiro).toBeDefined();
});
```

#### Backend (Jest + Supertest)
- **Unit tests:** Each agent returns valid JSON with required fields
- **Integration tests:** Orchestrator consolidates 3 agents correctly, handles partial failures
- **Route tests:** Auth returns 401 without token, 200 with valid token; /api/validar persists to DB
- **Database tests:** Prisma schema validates, migrations apply cleanly
- **Error scenarios:** Network timeout, malformed JSON from agent, invalid input validation
- **Service tests:** OpenAI wrapper retries, timeout handling, token tracking

#### Frontend (Jest + React Testing Library)
- **Component tests:** Login renders form, submit calls API, error displays
- **Hook tests:** useValidacao state transitions, API call handling
- **Integration tests:** Form submit → API call → dashboard render
- **Error UI:** Loading states, timeout messages, partial failure handling

#### E2E (Playwright)
- **User workflows:**
  - Login → submit idea → view dashboard → check history
  - Edit idea → submit → verify consolidation updated
  - Delete validation → confirm removal from history
- **Error paths:**
  - Expired token → logout + redirect to login
  - Network timeout → show retry button
  - Invalid input → validation error appears
- **Performance:** Dashboard loads within 3s, history pagination works

## Key Decisions & Constraints

| Decision | Rationale | Impact |
|----------|-----------|--------|
| **Partial agent failure returns partial consolidation, not 500** | UX: show partial analysis instead of blank; user still gets value | Error response includes `"erro": "timeout"` per agent |
| **gpt-4o-mini in dev, gpt-4o for production** | Cost control; easy to swap later | Update `OPENAI_MODEL` env var; same prompt formats |
| **SQLite for MVP** | No DevOps overhead; good for prototyping | Migrate to PostgreSQL if scaling beyond 10k users |
| **JWT with email-only login** | Simplicity for MVP; no password reset logic | Upgrade to email+password or OAuth later |
| **Frozen API contract in Phase 0** | Prevent mid-implementation rewrites | Breaking changes require team sync + simultaneous updates |
| **Backend-first, frontend after Phase 3** | Unblock frontend dev on stable API | Frontend can mock responses while backend is being built |
| **Semantic commits from all members** | Rubric requirement; traceability | Use commit hooks (husky) if needed to enforce |

## Common Patterns & Anti-Patterns

### BDD Best Practices

✅ **Good Gherkin Scenarios:**
- One scenario = one behavior (not 5 behaviors per scenario)
- Use concrete values: "entrepreneur@example.com" not "<email>"
- Describe user intent, not implementation: "user submits form" not "onClick fires"
- Use data tables for complex inputs
- Include side effects: DB persist, token stored, cache cleared
- Link to acceptance criteria: `# User Story #1 AC 2`

❌ **Bad Gherkin Scenarios:**
- UI-centric: focusing on button positions instead of behavior
- Overly detailed: too many Givens = brittle tests
- Mixed concerns: multiple user journeys in one scenario
- Implementation details: testing internal state instead of outcomes
- Missing error cases: only happy path tested

### ✅ DO
- **Write Gherkin FIRST** (BDD: behavior before code, always)
- **Write tests alongside code** (TDD mindset: BDD → unit → E2E)
- **Map scenarios to acceptance criteria** (User Story → AC → Gherkin)
- **Run all test layers** (`npm run test:all`: BDD + unit + E2E)
- **Commit frequently with semantic messages** (helps traceability)
- **Document API in Swagger/JSDoc** (single source of truth)
- **Small, focused PRs** (1-2 features per PR)
- **Update docs immediately** when code changes schema/API
- **Ask for help in standup** if blocked >30 min
- **Use semantic branch names** (`feature/agents`, `bugfix/xyz`)
- **Test error paths** (timeouts, invalid responses, network failures)
- **Include Gherkin file link in PR** ("See features/validation.feature")
- **Use .env for secrets** (OPENAI_API_KEY, JWT_SECRET, DATABASE_URL)
- **Validate input** on both client and server (never trust frontend)

### ❌ DON'T
- **Write code before Gherkin** (BDD: behavior first, code second)
- **Ship code without BDD+Jest+E2E tests** (all 3 layers required)
- **Skip acceptance criteria in scenarios** (every AC needs a scenario)
- **Use vague commit messages** (`fixed stuff`, `WIP`, `asdasd`)
- **Test only happy path** (test errors, timeouts, edge cases)
- **Merge without approval** (even if you're the lead)
- **Hardcode secrets** in code or version control
- **Skip Swagger documentation** for new endpoints
- **Change API contract without team discussion**
- **Ignore coverage reports** (monitor baseline)
- **Update Gherkin without updating tests** (keep in sync)
- **Leave TODO comments** without issue references
- **Force-push to develop** (breaks colleague history)

## Definition of Done (per task)

A task is **done** when:

1. ✅ **Gherkin scenarios written first** (BDD: .feature file with Given-When-Then)
2. ✅ **Scenarios pass** (`npm run test:bdd` exits 0)
3. ✅ **Code is implemented** (feature branch → PR)
4. ✅ **Unit tests passing** (`npm test`, >=30% coverage for that module)
5. ✅ **E2E tests passing** (`npm run test:e2e` succeeds)
6. ✅ **No API contract breakage** (or explicit contract update + notification)
7. ✅ **No regressions** (manual smoke test + run existing test suite)
8. ✅ **Docs are updated** (README, docs/*, comments, Gherkin linked in PR)
9. ✅ **PR is reviewed** (minimum 1 approval from module owner; BDD verified)
10. ✅ **Merged to develop** (and verified in develop branch)
11. ✅ **Ready for demo** (can show to product owner with Gherkin scenarios as test evidence)
